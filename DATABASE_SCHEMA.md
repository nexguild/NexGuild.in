## 1. Schema Overview

NexGuild V1 uses a single PostgreSQL schema (`public`) on Supabase. Authentication is handled by Supabase Auth (`auth.users`). The `public.users` table extends auth identities with platform-specific profile and wallet data.

All monetary values are stored as `NUMERIC(12,4)` — four decimal places of precision in USD. No floating-point types are used for money.

All primary keys are `UUID` generated by `gen_random_uuid()`. Supabase auth IDs are also UUIDs, making cross-table joins consistent.

All timestamps are `TIMESTAMPTZ` (timezone-aware), stored in UTC.

```
┌────────────────────────────────────────────────────────────┐
│                     SCHEMA RELATIONSHIPS                    │
│                                                            │
│  auth.users ──────────── public.users                      │
│                               │                            │
│                    ┌──────────┼──────────────┐             │
│                    │          │              │             │
│              opportunities  projects   submissions          │
│                    │          │              │             │
│                    └──────────┤              │             │
│                          project_tasks ──────┘             │
│                                                            │
│  users ──────────────── wallet_transactions                 │
│  users ──────────────── withdrawal_requests                 │
│  users ──────────────── admin_notes (target or author)     │
│  users ──────────────── notifications                       │
│                                                            │
│  offerwall_providers ─── offerwall_transactions            │

│                                                            │
│  platform_settings  (standalone key-value store)           │
└────────────────────────────────────────────────────────────┘
```

## 2. Design Decisions

### 2.1 Single User Table for Contributors and Admins
Rather than separate tables, all users (contributors and admins) live in `public.users` with a `role` column (`contributor` | `admin`). This simplifies foreign key references across the schema — every action referencing a user uses the same `users.id`. Row-Level Security (RLS) policies enforce what each role can read and write.

### 2.2 Wallet Balances Stored on the User Row
`wallet_available_balance` and `wallet_pending_balance` are denormalized onto the `users` row. This enables fast balance reads without aggregating the full `wallet_transactions` ledger every time. The `wallet_transactions` table is the source of truth — user balances must always equal the sum of confirmed transactions. Any balance update atomically writes both the transaction record and the user balance in a database transaction.

### 2.3 Immutable Ledger
`wallet_transactions` rows are never updated or deleted. Each balance change creates a new row. This provides a complete, auditable history. The `balance_after` snapshot on each row allows point-in-time balance reconstruction without replaying all transactions.

### 2.4 Polymorphic Source References on Wallet Transactions
`wallet_transactions.source_id` references different tables depending on `source_type` (submission, offerwall transaction, withdrawal request, or admin adjustment). PostgreSQL does not natively enforce polymorphic foreign keys, so this is enforced at the application layer. The `source_type` field always clarifies what `source_id` points to.

### 2.5 Payout Snapshot on Submissions
`submissions.payout_amount` captures the opportunity's payout at submission time. This prevents retroactive payout changes if an admin later edits the opportunity's payout value. The snapshot is the authoritative payout for that submission.

### 2.6 Project Tasks as First-Class Records
Managed project tasks live in `project_tasks` (a child of `projects`), separate from standalone `opportunities`. A submission can reference either an `opportunity_id` or a `project_task_id` — never both. This keeps the submission model flexible without merging two structurally different work types.

### 2.7 Payout Details as JSONB on Withdrawals
Withdrawal payout details (PayPal email, crypto address, network) vary by method. Storing as `JSONB` accommodates the different shapes without null-heavy sparse columns. The application validates structure based on the `method` value before writing.

## 3. Core Tables

### 3.1 `users`

**Purpose:** Stores all platform users — both contributors and admins. Extends Supabase Auth with profile data, wallet balances, account status, and platform-specific metadata. This is the central table that every other table references.

**Relationship to Supabase Auth:** `users.id` is the same UUID as `auth.users.id`. Created automatically via a database trigger when a new auth user registers.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | — | Primary key. Matches `auth.users.id` exactly. |
| `email` | `TEXT` | NOT NULL | — | User's email address. Unique across all users. Synced from `auth.users`. |
| `full_name` | `TEXT` | NOT NULL | — | Display name provided at registration. |
| `country_code` | `CHAR(2)` | NULL | `NULL` | ISO 3166-1 alpha-2 country code. Set at registration. Used for opportunity eligibility filtering. |
| `avatar_url` | `TEXT` | NULL | `NULL` | URL to the contributor's uploaded avatar image, stored in Supabase Storage. |
| `role` | `TEXT` | NOT NULL | `'contributor'` | User role. Allowed values: `contributor`, `admin`. Enforced by check constraint. |
| `status` | `TEXT` | NOT NULL | `'active'` | Account standing. Allowed values: `active`, `suspended`, `banned`. |
| `email_verified` | `BOOLEAN` | NOT NULL | `false` | Whether the user has verified their email address. Managed by Supabase Auth. |
| `is_identity_verified` | `BOOLEAN` | NOT NULL | `false` | Whether the user has completed identity verification for high-tier tasks. Manually set by admin. |
| `wallet_available_balance` | `NUMERIC(12,4)` | NOT NULL | `0.0000` | Confirmed, withdrawable balance. Denormalized for fast reads. Must reconcile with `wallet_transactions`. |
| `wallet_pending_balance` | `NUMERIC(12,4)` | NOT NULL | `0.0000` | Balance from approved-but-not-yet-confirmed earnings. Not withdrawable. |
| `wallet_frozen` | `BOOLEAN` | NOT NULL | `false` | When `true`, withdrawals are blocked. Set by admin during investigations. Does not affect earning. |
| `skills` | `TEXT[]` | NULL | `NULL` | Array of self-reported skill tags (e.g., `{"writing", "python", "translation"}`). Used for future project matching. |
| `total_submissions` | `INTEGER` | NOT NULL | `0` | Running count of all submissions ever made. Updated on each submission creation. |
| `approved_submissions` | `INTEGER` | NOT NULL | `0` | Running count of approved submissions. Used to compute approval rate. Updated on submission approval. |
| `total_earned` | `NUMERIC(12,4)` | NOT NULL | `0.0000` | Lifetime confirmed earnings. Updated on each wallet credit confirmation. Informational only — not used for balance. |
| `withdrawal_hold_until` | `TIMESTAMPTZ` | NULL | `NULL` | If set, no withdrawal can be submitted before this timestamp. Used for new account hold periods. |
| `last_active_at` | `TIMESTAMPTZ` | NULL | `NULL` | Timestamp of last authenticated action. Updated on dashboard loads. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Account creation timestamp. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last row modification timestamp. Maintained by trigger. |

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `auth.users.id → users.id` | One-to-one | Auth user always has exactly one profile row. |
| `users.id ← submissions.user_id` | One-to-many | A user has many submissions. |
| `users.id ← wallet_transactions.user_id` | One-to-many | A user has many ledger entries. |
| `users.id ← withdrawal_requests.user_id` | One-to-many | A user has many withdrawal requests. |
| `users.id ← notifications.user_id` | One-to-many | A user has many notifications. |
| `users.id ← admin_notes.target_id` | One-to-many | When `target_type = 'user'`. |
| `users.id ← admin_notes.admin_id` | One-to-many | Notes authored by this admin. |
| `users.id ← opportunities.created_by` | One-to-many | Admin who created the opportunity. |
| `users.id ← projects.created_by` | One-to-many | Admin who created the project. |
| `users.id ← project_tasks.assigned_to` | One-to-many | Contributor assigned to project tasks. |

---

#### Computed Values (Application Layer)

These are not stored columns but derived at read time:

- **Approval Rate:** `approved_submissions / total_submissions * 100` — shown on contributor profile.
- **Pending Withdrawal Count:** Count of `withdrawal_requests` where `user_id = id AND status = 'pending'`.

### 3.2 `opportunities`

**Purpose:** Stores all standalone work opportunities created by admins and made available to contributors. Does not include project tasks (those live in `project_tasks`). Covers opportunity types: survey, micro-task, data labeling, content task. Offerwall tasks do not create rows here — they are handled by the offerwall system.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `title` | `TEXT` | NOT NULL | — | Short display title shown on opportunity cards. |
| `description` | `TEXT` | NOT NULL | — | Public-facing description shown to contributors. |
| `instructions` | `TEXT` | NOT NULL | — | Full instructions shown to the contributor after starting the task. Supports markdown. |
| `type` | `TEXT` | NOT NULL | — | Opportunity category. Allowed values: `survey`, `micro_task`, `data_labeling`, `content_task`. |
| `payout_amount` | `NUMERIC(10,4)` | NOT NULL | — | Fixed payout per approved submission in USD. Must be greater than zero. |
| `estimated_minutes` | `INTEGER` | NULL | `NULL` | Estimated time to complete in minutes. Displayed to contributors for planning. |
| `skill_level` | `TEXT` | NOT NULL | `'any'` | Required skill level. Allowed values: `any`, `intermediate`, `advanced`. |
| `max_submissions` | `INTEGER` | NULL | `NULL` | Maximum number of approved submissions accepted. `NULL` means unlimited. |
| `total_submission_count` | `INTEGER` | NOT NULL | `0` | Running count of all submissions received (any status). |
| `approved_submission_count` | `INTEGER` | NOT NULL | `0` | Running count of approved submissions. Used to enforce `max_submissions`. |
| `status` | `TEXT` | NOT NULL | `'draft'` | Publication state. Allowed values: `draft`, `published`, `paused`, `archived`. |
| `requires_identity_verification` | `BOOLEAN` | NOT NULL | `false` | If `true`, only contributors with `is_identity_verified = true` can access. |
| `eligible_countries` | `TEXT[]` | NULL | `NULL` | Array of ISO alpha-2 country codes allowed to participate. `NULL` means all countries. |
| `allow_retry` | `BOOLEAN` | NOT NULL | `false` | Whether a contributor can resubmit after a rejection. |
| `max_retries` | `INTEGER` | NOT NULL | `1` | Maximum number of retry attempts allowed per contributor. Irrelevant if `allow_retry = false`. |
| `time_limit_minutes` | `INTEGER` | NULL | `NULL` | If set, submission must be made within this many minutes of starting. Enforced at submit time. |
| `deadline` | `TIMESTAMPTZ` | NULL | `NULL` | Hard deadline. Opportunity auto-pauses after this timestamp if not already archived. |
| `created_by` | `UUID` | NOT NULL | — | FK to `users.id`. Must reference a user with `role = 'admin'`. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Creation timestamp. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last modification timestamp. Maintained by trigger. |

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `opportunities.created_by → users.id` | Many-to-one | Admin who created and owns this opportunity. |
| `opportunities.id ← submissions.opportunity_id` | One-to-many | All submissions for this opportunity. |
| `opportunities.id ← admin_notes.target_id` | One-to-many | When `target_type = 'opportunity'`. |

---

#### Business Rules

- An opportunity with `status = 'archived'` is permanently closed. It cannot be republished.
- When `approved_submission_count` reaches `max_submissions`, the opportunity status must be set to `paused` automatically (application layer).
- A contributor may only have one active (non-rejected) submission per opportunity at a time. This is enforced by a unique partial index (see Index Recommendations).
- `payout_amount` must not be changed while the opportunity has any submissions in `submitted` or `approved` status, to preserve payout integrity.

---

#### Index Recommendations

| Index | Columns | Type | Reason |
|---|---|---|---|
| `opportunities_pkey` | `id` | Primary Key (B-tree) | All FK lookups. |
| `opportunities_status_idx` | `status` | B-tree | Contributor browse page — filter published only. |
| `opportunities_type_status_idx` | `(type, status)` | B-tree | Filter by type and published status together. |
| `opportunities_created_by_idx` | `created_by` | B-tree | Admin view of their own opportunities. |
| `opportunities_deadline_idx` | `deadline` | B-tree | Background job to auto-pause expired opportunities. |
| `opportunities_created_at_idx` | `created_at DESC` | B-tree | Admin list ordered by recency. |

### 3.3 `projects`

**Purpose:** Stores managed projects created by admins on behalf of organizations. A project is a container for related `project_tasks`. Projects are fully admin-managed — contributors interact only with the individual tasks assigned to them, not the project itself. Organization names are stored internally for admin reference only and never exposed to contributors.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `title` | `TEXT` | NOT NULL | — | Internal project title. Visible to admins and, in limited form, to assigned contributors. |
| `description` | `TEXT` | NOT NULL | — | Internal project description. Admin-only. |
| `type` | `TEXT` | NOT NULL | — | Project category. Allowed values: `data_collection`, `research_survey`, `content_production`, `qa_testing`, `data_annotation`. |
| `status` | `TEXT` | NOT NULL | `'draft'` | Project lifecycle state. Allowed values: `draft`, `active`, `completed`, `cancelled`. |
| `total_budget` | `NUMERIC(12,4)` | NOT NULL | — | Total USD budget allocated for contributor payouts on this project. Must be greater than zero. |
| `budget_allocated` | `NUMERIC(12,4)` | NOT NULL | `0.0000` | Sum of `payout_amount` across all `project_tasks` under this project. Must not exceed `total_budget`. Updated when tasks are added. |
| `budget_paid_out` | `NUMERIC(12,4)` | NOT NULL | `0.0000` | Sum of approved and paid out task payouts. Updated when submissions are approved. |
| `organization_name` | `TEXT` | NULL | `NULL` | Internal reference to the organization this project was created for. Never shown to contributors. |
| `brief` | `TEXT` | NULL | `NULL` | Full internal brief document. Admin-only. Supports markdown. |
| `internal_notes` | `TEXT` | NULL | `NULL` | Running internal notes field for admin use. Not structured. |
| `deadline` | `TIMESTAMPTZ` | NULL | `NULL` | Expected project completion date. For admin tracking only. |
| `created_by` | `UUID` | NOT NULL | — | FK to `users.id`. Must reference a user with `role = 'admin'`. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Creation timestamp. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last modification timestamp. Maintained by trigger. |

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `projects.created_by → users.id` | Many-to-one | Admin who owns this project. |
| `projects.id ← project_tasks.project_id` | One-to-many | All tasks under this project. |
| `projects.id ← admin_notes.target_id` | One-to-many | When `target_type = 'project'`. |

---

#### Business Rules

- `budget_allocated` must never exceed `total_budget`. Enforced at the application layer when adding tasks.
- A project in `cancelled` status must not have new tasks added or new submissions accepted.
- `organization_name` is internal only. RLS policies must prevent contributors from ever reading this column.

---

#### Index Recommendations

| Index | Columns | Type | Reason |
|---|---|---|---|
| `projects_pkey` | `id` | Primary Key (B-tree) | All FK lookups. |
| `projects_status_idx` | `status` | B-tree | Admin project list filtered by status. |
| `projects_created_by_idx` | `created_by` | B-tree | Admin sees their own projects. |
| `projects_created_at_idx` | `created_at DESC` | B-tree | Admin list ordered by recency. |

### 3.4 `project_tasks`

**Purpose:** Individual work units within a managed project. Each task can be assigned to one contributor and produces one submission. Project tasks are structurally similar to opportunities but are part of a managed workflow — assignment is controlled by admins, not contributor self-selection. Contributors see only their assigned tasks.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `project_id` | `UUID` | NOT NULL | — | FK to `projects.id`. The parent project. |
| `title` | `TEXT` | NOT NULL | — | Task title. Shown to the assigned contributor. |
| `description` | `TEXT` | NOT NULL | — | Task description. Shown to the assigned contributor. |
| `instructions` | `TEXT` | NOT NULL | — | Full task instructions. Supports markdown. Shown after assignment. |
| `payout_amount` | `NUMERIC(10,4)` | NOT NULL | — | USD payout for this specific task upon approval. Must be greater than zero. |
| `assigned_to` | `UUID` | NULL | `NULL` | FK to `users.id`. The contributor assigned to this task. `NULL` when unassigned. Must reference a user with `role = 'contributor'`. |
| `assigned_at` | `TIMESTAMPTZ` | NULL | `NULL` | Timestamp when the task was assigned to a contributor. |
| `status` | `TEXT` | NOT NULL | `'unassigned'` | Task lifecycle state. Allowed values: `unassigned`, `assigned`, `in_progress`, `submitted`, `approved`, `rejected`. |
| `created_by` | `UUID` | NOT NULL | — | FK to `users.id`. Admin who created this task. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Task creation timestamp. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last modification timestamp. Maintained by trigger. |

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `project_tasks.project_id → projects.id` | Many-to-one | Parent project. Cascade delete is intentionally NOT set — tasks should be manually archived. |
| `project_tasks.assigned_to → users.id` | Many-to-one | Assigned contributor. Nullable. |
| `project_tasks.created_by → users.id` | Many-to-one | Admin who created this task. |
| `project_tasks.id ← submissions.project_task_id` | One-to-one | Each task produces at most one active submission. |

---

#### Business Rules

- A task may only be assigned to one contributor at a time.
- When a task is assigned (`assigned_to` set), `status` transitions to `assigned`.
- When the contributor starts work, `status` transitions to `in_progress`.
- A `project_tasks` row transitions to `submitted` when a linked submission is created.
- `assigned_to` must always reference a user with `role = 'contributor'`.
- Adding a task to a project must update `projects.budget_allocated` by the task's `payout_amount`.

---

#### Index Recommendations

| Index | Columns | Type | Reason |
|---|---|---|---|
| `project_tasks_pkey` | `id` | Primary Key (B-tree) | All FK lookups. |
| `project_tasks_project_id_idx` | `project_id` | B-tree | Load all tasks for a project. |
| `project_tasks_assigned_to_idx` | `assigned_to` | B-tree | Contributor's "My Tasks" view for project tasks. |
| `project_tasks_status_idx` | `status` | B-tree | Admin filtering by status. |
| `project_tasks_project_status_idx` | `(project_id, status)` | B-tree | Admin project detail view — tasks by status. |

### 3.5 `submissions`

**Purpose:** Records every submission a contributor makes, for either a standalone opportunity or a project task. A submission carries the content the contributor provided, its current review status, the payout amount (snapshot), and admin review metadata. This table is the central queue for admin review and drives wallet credit events on approval.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `user_id` | `UUID` | NOT NULL | — | FK to `users.id`. The contributor who made this submission. |
| `opportunity_id` | `UUID` | NULL | `NULL` | FK to `opportunities.id`. Set if this is a standalone opportunity submission. Mutually exclusive with `project_task_id`. |
| `project_task_id` | `UUID` | NULL | `NULL` | FK to `project_tasks.id`. Set if this is a project task submission. Mutually exclusive with `opportunity_id`. |
| `content` | `JSONB` | NOT NULL | — | The actual submission content. Structure varies by opportunity type. See Content Schema Reference below. |
| `content_type` | `TEXT` | NOT NULL | — | Describes the shape of `content`. Allowed values: `text`, `url`, `file_urls`, `form_responses`, `mixed`. |
| `status` | `TEXT` | NOT NULL | `'in_progress'` | Review lifecycle state. Allowed values: `in_progress`, `submitted`, `approved`, `rejected`. |
| `payout_amount` | `NUMERIC(10,4)` | NOT NULL | — | Snapshot of the payout at submission time. Taken from `opportunities.payout_amount` or `project_tasks.payout_amount`. Immutable after creation. |
| `feedback` | `TEXT` | NULL | `NULL` | Admin feedback message. Set on rejection. Displayed to the contributor. |
| `reviewed_by` | `UUID` | NULL | `NULL` | FK to `users.id`. Admin who approved or rejected this submission. |
| `reviewed_at` | `TIMESTAMPTZ` | NULL | `NULL` | Timestamp of the review action. |
| `started_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | When the contributor clicked "Start Task." Used for time-limit enforcement. |
| `submitted_at` | `TIMESTAMPTZ` | NULL | `NULL` | When the contributor submitted the form. `NULL` while in `in_progress`. |
| `attempt_number` | `INTEGER` | NOT NULL | `1` | Which attempt this is. `1` for first submission, increments on retry. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Row creation timestamp. Same as `started_at` in practice. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last modification timestamp. Maintained by trigger. |

#### Content JSONB Schema Reference

The `content` field stores different shapes depending on `content_type`:

**`text`** — Plain written response:
```json
{
  "body": "The full text of the submission..."
}
```

**`url`** — A link submission (e.g., published article):
```json
{
  "url": "https://...",
  "note": "Optional note from contributor"
}
```

**`file_urls`** — One or more uploaded file references (Supabase Storage paths):
```json
{
  "files": [
    { "name": "image1.jpg", "path": "submissions/uuid/image1.jpg" },
    { "name": "audio.mp3", "path": "submissions/uuid/audio.mp3" }
  ]
}
```

**`form_responses`** — Structured key-value answers to a defined question set:
```json
{
  "responses": [
    { "question_id": "q1", "answer": "Option A" },
    { "question_id": "q2", "answer": "4" }
  ]
}
```

**`mixed`** — Combination of the above:
```json
{
  "body": "Written notes...",
  "files": [...],
  "url": "https://..."
}
```

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `submissions.user_id → users.id` | Many-to-one | Contributor who submitted. |
| `submissions.opportunity_id → opportunities.id` | Many-to-one | Nullable. Standalone opportunity link. |
| `submissions.project_task_id → project_tasks.id` | Many-to-one | Nullable. Project task link. |
| `submissions.reviewed_by → users.id` | Many-to-one | Nullable. Admin reviewer. |
| `submissions.id ← wallet_transactions.source_id` | One-to-one | When `source_type = 'submission_approval'`. |
| `submissions.id ← admin_notes.target_id` | One-to-many | When `target_type = 'submission'`. |

---

#### Business Rules

- Exactly one of `opportunity_id` or `project_task_id` must be set. Both cannot be null; both cannot be non-null. Enforced by a check constraint.
- A contributor may not have more than one submission in `in_progress` or `submitted` status for the same `opportunity_id`. Enforced by a unique partial index.
- When status changes to `approved`, a `wallet_transactions` credit record must be created in the same database transaction.
- `payout_amount` is copied at submission creation and never updated afterward.
- When `attempt_number > 1`, there must exist a prior `rejected` submission from the same user for the same opportunity.
- File paths stored in `content` (for `file_urls` type) must reference objects in Supabase Storage. Deletion of storage objects must not happen before the submission is archived.

### 3.6 `wallet_transactions`

**Purpose:** The complete, immutable financial ledger for every contributor's wallet. Every credit and debit — from task approval, offerwall earnings, withdrawal execution, or admin adjustment — creates a row here. Rows are never updated or deleted. The running `balance_after` snapshot on each row enables balance reconstruction at any point in time without replaying all transactions. This table is the source of truth for all money movement.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `user_id` | `UUID` | NOT NULL | — | FK to `users.id`. The contributor whose wallet is affected. |
| `type` | `TEXT` | NOT NULL | — | Direction of the transaction. Allowed values: `credit`, `debit`. |
| `source_type` | `TEXT` | NOT NULL | — | What caused this transaction. Allowed values: `submission_approval`, `offerwall`, `withdrawal`, `admin_adjustment`. |
| `source_id` | `UUID` | NULL | `NULL` | Polymorphic reference to the causative record. Points to `submissions.id`, `offerwall_transactions.id`, `withdrawal_requests.id`, or `NULL` for admin manual adjustments. |
| `amount` | `NUMERIC(10,4)` | NOT NULL | — | Absolute transaction amount in USD. Always positive. Direction is determined by `type`. Must be greater than zero. |
| `balance_type` | `TEXT` | NOT NULL | — | Which balance bucket this transaction affects. Allowed values: `pending`, `available`. |
| `balance_after` | `NUMERIC(12,4)` | NOT NULL | — | Snapshot of the `balance_type` balance after this transaction is applied. Enables point-in-time reconstruction. |
| `description` | `TEXT` | NOT NULL | — | Human-readable description shown in contributor earnings history. Examples: "Survey completed: Market Research Study", "Withdrawal via PayPal", "Lootably offerwall earnings". |
| `created_by` | `UUID` | NULL | `NULL` | FK to `users.id`. Set only for `admin_adjustment` type transactions, referencing the admin who made the adjustment. `NULL` for all system-generated transactions. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Transaction timestamp. Immutable after creation. |

---

#### Typical Transaction Flows

**Task approval — two transactions:**
1. `credit` | `source_type = submission_approval` | `balance_type = pending` — Earnings added to pending on approval.
2. `credit` | `source_type = submission_approval` | `balance_type = available` + corresponding `debit` from `pending` — Pending balance converts to available after settlement period.

*Note: In V1, the settlement conversion may be triggered manually by admin or on a scheduled basis. The two-step flow preserves the balance model even if settlement is near-instant initially.*

**Offerwall earnings — one transaction:**
1. `credit` | `source_type = offerwall` | `balance_type = available` — Offerwall earnings are credited directly to available (no pending period, as provider confirmation is the equivalent).

**Withdrawal — one transaction:**
1. `debit` | `source_type = withdrawal` | `balance_type = available` — Available balance reduced when withdrawal is marked Completed.

**Admin adjustment — one or two transactions:**
- Manual credit: `credit` | `source_type = admin_adjustment` | `balance_type = available` (or `pending`).
- Manual debit: `debit` | `source_type = admin_adjustment` | `balance_type = available`.

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `wallet_transactions.user_id → users.id` | Many-to-one | Wallet owner. |
| `wallet_transactions.created_by → users.id` | Many-to-one | Nullable. Admin for manual adjustments. |
| `wallet_transactions.source_id` | Polymorphic | Application-enforced: references `submissions`, `offerwall_transactions`, or `withdrawal_requests`. |

---

#### Business Rules

- Rows in this table are **immutable**. No `UPDATE` or `DELETE` operations are permitted. Application-layer and RLS policies must enforce this.
- Every write to `wallet_transactions` must be paired with an update to `users.wallet_available_balance` or `users.wallet_pending_balance` in the same database transaction.
- `balance_after` must be calculated before insertion and must reflect the actual balance after the transaction. The application must hold a row lock on the user's balance during this calculation to prevent race conditions.
- `amount` must always be greater than zero.
- A debit must never reduce a balance below zero. Enforced at the application layer before creating the transaction row.

---

#### Index Recommendations

| Index | Columns | Type | Reason |
|---|---|---|---|
| `wallet_transactions_pkey` | `id` | Primary Key (B-tree) | Standard. |
| `wallet_transactions_user_id_idx` | `user_id` | B-tree | Load all transactions for a contributor. |
| `wallet_transactions_user_created_idx` | `(user_id, created_at DESC)` | B-tree | Contributor earnings history, time-ordered. |
| `wallet_transactions_source_idx` | `(source_type, source_id)` | B-tree | Look up transaction by source record. |
| `wallet_transactions_created_by_idx` | `created_by` | B-tree | Audit admin manual adjustments. |
| `wallet_transactions_created_at_idx` | `created_at DESC` | B-tree | Admin finances view — all transactions by time. |

### 3.7 `withdrawal_requests`

**Purpose:** Records contributor requests to withdraw available wallet balance via a supported payment method. Each request progresses through a status lifecycle managed by admins. Payment execution happens externally (PayPal dashboard, crypto transfer); this table tracks the request, admin processing status, and links back to the wallet debit transaction on completion.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `user_id` | `UUID` | NOT NULL | — | FK to `users.id`. The contributor requesting the withdrawal. |
| `amount` | `NUMERIC(10,4)` | NOT NULL | — | Requested withdrawal amount in USD. Must be ≥ the platform minimum. Must not exceed the contributor's `wallet_available_balance` at request time. |
| `method` | `TEXT` | NOT NULL | — | Payment method selected. Allowed values: `paypal`, `bitcoin`, `usdt`. |
| `payout_details` | `JSONB` | NOT NULL | — | Payment destination details. Structure varies by method. See Payout Details Schema below. |
| `status` | `TEXT` | NOT NULL | `'pending'` | Request lifecycle state. Allowed values: `pending`, `processing`, `completed`, `rejected`. |
| `rejection_reason` | `TEXT` | NULL | `NULL` | Reason provided by admin when rejecting. Required for rejections — enforced at application layer. |
| `wallet_transaction_id` | `UUID` | NULL | `NULL` | FK to `wallet_transactions.id`. Set when the withdrawal is completed and the debit transaction is written. |
| `processed_by` | `UUID` | NULL | `NULL` | FK to `users.id`. Admin who last acted on this request. |
| `processing_started_at` | `TIMESTAMPTZ` | NULL | `NULL` | Timestamp when admin marked status `processing`. |
| `completed_at` | `TIMESTAMPTZ` | NULL | `NULL` | Timestamp when admin marked status `completed`. |
| `created_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | When the contributor submitted the request. |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Last status change timestamp. Maintained by trigger. |

---

#### Payout Details JSONB Schema

**PayPal:**
```json
{
  "paypal_email": "contributor@example.com"
}
```

**Bitcoin:**
```json
{
  "crypto_address": "bc1q...",
  "network": "bitcoin"
}
```

**USDT:**
```json
{
  "crypto_address": "0x...",
  "network": "trc20"
}
```

*Note: `payout_details` stores exactly what the contributor entered. The application must validate the shape against the method before writing. Sensitive address data stored here is readable by admins only — contributor RLS policy restricts visibility of this column to their own rows.*

---

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `withdrawal_requests.user_id → users.id` | Many-to-one | Contributor who requested the withdrawal. |
| `withdrawal_requests.processed_by → users.id` | Many-to-one | Nullable. Admin who processed it. |
| `withdrawal_requests.wallet_transaction_id → wallet_transactions.id` | One-to-one | Nullable. Linked debit transaction on completion. |
| `withdrawal_requests.id ← admin_notes.target_id` | One-to-many | When `target_type = 'withdrawal_request'`. |

---

#### Business Rules

- A contributor may have at most one request in `pending` or `processing` status at any time. A new request cannot be submitted while one is open. Enforced by a unique partial index.
- On submission, the request amount must be atomically reserved — `users.wallet_available_balance` is reduced immediately and a pending debit intent is recorded. This prevents double-spending if a second request is attempted concurrently. If the request is rejected, the reserved amount is restored.
- Alternatively (simpler V1 approach): block new requests while one is open (the unique partial index below handles this) and only debit the balance on `completed`.
- `rejection_reason` is required when setting status to `rejected`. Enforced at application layer.
- `wallet_transaction_id` must be set in the same database transaction as the status change to `completed`.
- A request in `completed` or `rejected` status cannot change status again.

---

#### Index Recommendations

| Index | Columns | Type | Reason |
|---|---|---|---|
| `withdrawal_requests_pkey` | `id` | Primary Key (B-tree) | Standard. |
| `withdrawal_requests_user_id_idx` | `user_id` | B-tree | Contributor's withdrawal history. |
| `withdrawal_requests_status_idx` | `status` | B-tree | Admin queue filtered by `pending`. |
| `withdrawal_requests_processed_by_idx` | `processed_by` | B-tree | Admin audit of who processed what. |
| `withdrawal_requests_created_at_idx` | `created_at DESC` | B-tree | Admin queue ordered by request date. |
| `withdrawal_requests_user_open_uidx` | `user_id` WHERE `status IN ('pending', 'processing')` | Unique Partial (B-tree) | Enforce one open request per contributor at a time. |

### 4.2 `offerwall_transactions`

**Purpose:** Immutable record of every postback received from offerwall providers. Serves two purposes: (1) idempotency guard — prevents duplicate wallet credits if the same postback is delivered more than once, and (2) audit trail — links every offerwall wallet credit back to a specific provider event. Each validated, non-duplicate row triggers one `wallet_transactions` credit.

---

#### Columns

| Column | Type | Nullable | Default | Description |
|---|---|---|---|---|
| `id` | `UUID` | NOT NULL | `gen_random_uuid()` | Primary key. |
| `provider_id` | `UUID` | NOT NULL | — | FK to `offerwall_providers.id`. Which provider sent this postback. |
| `user_id` | `UUID` | NOT NULL | — | FK to `users.id`. Contributor credited. Resolved from the `user_id` parameter in the postback. |
| `provider_transaction_id` | `TEXT` | NOT NULL | — | The transaction identifier sent by the provider. Must be unique per provider. Used for idempotency. |
| `gross_amount` | `NUMERIC(10,4)` | NOT NULL | — | Full USD amount sent by the provider before NexGuild's share is deducted. |
| `contributor_amount` | `NUMERIC(10,4)` | NOT NULL | — | Amount credited to the contributor after applying `contributor_share_percent`. |
| `platform_amount` | `NUMERIC(10,4)` | NOT NULL | — | `gross_amount - contributor_amount`. NexGuild's retained revenue on this transaction. |
| `status` | `TEXT` | NOT NULL | `'received'` | Processing state. Allowed values: `received`, `credited`, `duplicate`, `invalid`. |
| `wallet_transaction_id` | `UUID` | NULL | `NULL` | FK to `wallet_transactions.id`. Set after the wallet credit is applied. `NULL` if duplicate or invalid. |
| `raw_payload` | `JSONB` | NULL | `NULL` | Full raw postback payload from the provider. Stored for debugging. |
| `received_at` | `TIMESTAMPTZ` | NOT NULL | `now()` | Timestamp the postback was received by NexGuild's endpoint. |

#### Relationships

| Relationship | Type | Notes |
|---|---|---|
| `offerwall_transactions.provider_id → offerwall_providers.id` | Many-to-one | Which provider sent the postback. |
| `offerwall_transactions.user_id → users.id` | Many-to-one | Contributor to credit. |
| `offerwall_transactions.wallet_transaction_id → wallet_transactions.id` | One-to-one | Nullable. Linked credit entry. |

#### Business Rules

- The combination of `(provider_id, provider_transaction_id)` must be unique. A second postback with the same pair sets `status = 'duplicate'` and does not create a wallet credit.
- `contributor_amount` is calculated at receipt time using the provider's current `contributor_share_percent`. The calculation is: `gross_amount * (contributor_share_percent / 100)`, rounded to 4 decimal places.

## 7. Index Strategy

### 7.1 Primary Keys
Every table uses a `UUID` primary key. UUIDs avoid sequential ID guessing (security benefit) and are globally unique across tables (useful for polymorphic references). Supabase generates these via `gen_random_uuid()`.

### 7.2 Unique Constraints
- `users.email` — one account per email address.
- `offerwall_providers.slug` — machine-readable identifiers must be unique.
- `(offerwall_transactions.provider_id, offerwall_transactions.provider_transaction_id)` — core idempotency guard.

### 7.3 Unique Partial Indexes
These enforce business constraints that standard unique constraints cannot express:

- **`submissions`: one active submission per contributor per opportunity** — `UNIQUE (user_id, opportunity_id) WHERE status IN ('in_progress', 'submitted')`. A contributor can have one rejected and one new active submission if retries are allowed.
- **`withdrawal_requests`: one open request per contributor at a time** — `UNIQUE (user_id) WHERE status IN ('pending', 'processing')`.

### 7.4 Composite Indexes
Composite indexes are created for columns frequently queried together. Order of columns matters — put the most selective filter first, then the sort column:
- `(user_id, created_at DESC)` on `wallet_transactions` — contributor history view sorts by time within a user.
- `(project_id, status)` on `project_tasks` — admin project detail filters tasks by status.
- `(type, status)` on `opportunities` — browse page filters by both simultaneously.
- `(target_type, target_id)` on `admin_notes` — load all notes for a specific entity.

### 7.5 Partial Indexes
Partial indexes on boolean conditions reduce index size significantly for sparse flags:
- `offerwall_providers WHERE is_active = true` — only a small subset will ever be active.
- `notifications WHERE is_read = false` — most notifications will be read eventually; the unread set is small.

### 7.6 What Not to Index
- `wallet_transactions` rows are insert-only and high-volume. Do not add more indexes than necessary — each index increases write cost.
- JSONB columns (`content`, `payout_details`, `raw_payload`) should not be indexed in V1. If specific JSONB key queries become necessary post-launch, GIN indexes can be added selectively.
- Columns with very low cardinality (like `role` on `users` with only two values) are not good standalone index candidates. Use composite indexes that include a more selective leading column.

## 8. Row-Level Security Strategy

Supabase RLS is enabled on all tables. The following describes the intent of each table's policies. Exact policy syntax is defined at implementation time.

### 8.1 `users`
- **Contributors** can `SELECT` their own row only. Can `UPDATE` their own row for limited columns: `full_name`, `avatar_url`, `skills`. Cannot update `role`, `status`, `wallet_*`, `is_identity_verified`.
- **Admins** can `SELECT` all rows. Can `UPDATE` all columns.
- **System (service role)** can do all operations. Used for server-side triggers and functions.
- `organization_name` in `projects` — not applicable to `users`, but the equivalent sensitive columns must be blocked from contributor reads via column-level policy or view.

### 8.2 `opportunities`
- **Contributors** can `SELECT` only rows where `status = 'published'` and they meet eligibility criteria (`country_code` in `eligible_countries` or `eligible_countries IS NULL`). Cannot `INSERT`, `UPDATE`, or `DELETE`.
- **Admins** can do all operations.

### 8.3 `projects`
- **Contributors** cannot read `projects` rows directly. They interact with assigned `project_tasks` only.
- **Admins** can do all operations.

### 8.4 `project_tasks`
- **Contributors** can `SELECT` only rows where `assigned_to = auth.uid()`. Can `UPDATE` only the `status` column (to transition `assigned` → `in_progress`). Cannot `INSERT` or `DELETE`.
- **Admins** can do all operations.

### 8.5 `submissions`
- **Contributors** can `SELECT` their own submissions only (`user_id = auth.uid()`). Can `INSERT` new submissions. Can `UPDATE` `content` and `status` (for `in_progress` → `submitted` transition) only while `status = 'in_progress'`.
- **Admins** can do all operations.

### 8.6 `wallet_transactions`
- **Contributors** can `SELECT` their own transactions only. Cannot `INSERT`, `UPDATE`, or `DELETE` (all writes are via server-side functions or service role).
- **Admins** can `SELECT` all. No direct `INSERT`/`UPDATE`/`DELETE` — must go through defined functions.

### 8.7 `withdrawal_requests`
- **Contributors** can `SELECT` their own requests. Can `INSERT` new requests when conditions are met. Cannot `UPDATE` or `DELETE`.
- **Admins** can do all operations.

### 8.8 `admin_notes`
- **Contributors** have zero access (no `SELECT`, no `INSERT`).
- **Admins** can `SELECT` all and `INSERT`. No `UPDATE` or `DELETE` — immutable audit trail.

### 8.9 `offerwall_providers`
- **Contributors** can `SELECT` limited columns only: `id`, `name`, `slug`, `is_active`, `embed_url_template`. Cannot read `api_key`, `postback_secret`, `contributor_share_percent`, `notes`.
- **Admins** can do all operations.

### 8.10 `offerwall_transactions`
- **Contributors** can `SELECT` their own rows only (`user_id = auth.uid()`). Limited columns — no `raw_payload`, `gross_amount`, or `platform_amount`.
- **Admins** can do all operations.

### 8.11 `platform_settings`
- **Contributors** cannot read this table.
- **Admins** can `SELECT` and `UPDATE`. No `INSERT` or `DELETE` — rows are seeded at setup and updated in place.

### 8.12 `notifications`
- **Contributors** can `SELECT` and `UPDATE` (`is_read` column only) their own rows.
- **Admins** can do all operations. Cannot be sent contributor notifications through this table.

## 9. Data Integrity Rules

### 9.1 Financial Integrity
- The sum of all `wallet_transactions` credits minus debits for a given `user_id` and `balance_type` must always equal the corresponding `users.wallet_available_balance` or `users.wallet_pending_balance`. Any code path that creates a `wallet_transactions` row must update the user balance in the same transaction.
- No balance may go below zero. This must be checked and enforced before creating any debit transaction.
- `wallet_transactions` rows are write-once. Any correction requires an opposing transaction (credit to reverse a debit, or debit to reverse a credit), never an `UPDATE`.

### 9.2 Submission Integrity
- A submission must reference either `opportunity_id` or `project_task_id`, never both, never neither. Enforced by: `CHECK ((opportunity_id IS NOT NULL AND project_task_id IS NULL) OR (opportunity_id IS NULL AND project_task_id IS NOT NULL))`.
- `payout_amount` on a submission is set at creation and never changed. The value comes from the opportunity or project task at the time the contributor submits, not at review time.
- Approving a submission must create a `wallet_transactions` record in the same database transaction. If either operation fails, both must roll back.

### 9.3 Withdrawal Integrity
- A withdrawal can only be created when `users.wallet_available_balance >= amount` and `users.wallet_frozen = false` and `users.wallet_available_balance >= platform_settings.min_withdrawal_usd`.
- Completing a withdrawal must decrement `users.wallet_available_balance` and create a debit `wallet_transactions` record in the same transaction.
- Rejecting a withdrawal that had its balance reserved must restore the reserved amount atomically.

### 9.4 Referential Integrity
- All foreign keys must use `ON DELETE RESTRICT` by default. Rows cannot be deleted if other rows reference them. Soft deletion via `status` columns is preferred over hard deletes for operational data.
- The only exception: `notifications` may use `ON DELETE CASCADE` from `users`, as notifications have no referential value if the user is deleted.

### 9.5 Budget Integrity
- `projects.budget_allocated` must equal the sum of `project_tasks.payout_amount` for all tasks under the project. Updated synchronously when tasks are added or removed.
- `projects.budget_paid_out` must equal the sum of all `approved` project task payouts. Updated when submissions on project tasks are approved.
- `budget_allocated` must never exceed `total_budget`. Enforced at application layer before adding tasks.

## 10. Future Schema Considerations

These are not V1 requirements but are noted here to ensure V1 schema choices do not foreclose future expansion.

### 10.1 Contributor Tiers (V2)
A `contributor_tiers` table with `name`, `min_approved_submissions`, `min_approval_rate`, `benefits` (JSONB). A `tier_id` foreign key on `users`. Opportunities gain a `required_tier_id` column. The current `skill_level` column on `opportunities` may be deprecated in favor of tier requirements.

### 10.2 Organization Portal (V2)
A new `organizations` table separate from `users`. A new `organization_users` join table. Projects gain an `organization_id` FK. The `organization_name` text column on `projects` becomes a proper FK reference.

### 10.3 Referral Program (V2)
A `referrals` table: `referrer_id` (FK users), `referred_id` (FK users), `status`, `bonus_paid_at`. A `referred_by` column on `users`. Referral bonus credits go through `wallet_transactions` with a new `source_type = 'referral'`.

### 10.4 Blog / CMS (V1 Optional)
A `blog_posts` table: `id`, `slug`, `title`, `body` (markdown), `excerpt`, `tags` (text[]), `status` (`draft` | `published`), `published_at`, `author_id` (FK users with admin role). No contributor read permissions required until the blog is public.

### 10.5 Opportunity Questions (V2)
A `opportunity_questions` table to define structured question sets for `form_responses` submissions: `id`, `opportunity_id`, `question_text`, `type` (`text`, `select`, `number`, `boolean`), `options` (JSONB for select type), `order`, `required`. Replaces the current freeform JSONB approach for structured tasks.

### 10.6 Audit Log Table
A dedicated `audit_log` table for all schema-level changes beyond `admin_notes`: `id`, `table_name`, `row_id`, `operation` (`INSERT`|`UPDATE`|`DELETE`), `changed_by`, `old_data` (JSONB), `new_data` (JSONB), `created_at`. Populated by PostgreSQL triggers. Not required for V1 but strongly recommended before the platform reaches any meaningful scale.
